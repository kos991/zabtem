# Desktop Skeleton Supervision Design

Date: 2026-05-30

## Purpose

Implement the first development-plan step for Zabtem: a real Electron + React + TypeScript + Vite desktop application skeleton under `apps/desktop`, supervised by Claude and generated by a Codex subtask. This is only the desktop shell foundation. It must not add SNMP, SQLite, project CRUD, MIB parsing, or template business logic.

## Source Rules

This work follows the repository plan in `docs/plans/2026-05-30-zabbix-template-generator-plan.md` and `docs/plans/2026-05-30-zabbix-template-generator-design-notes.md`.

Mandatory project rules:

- Formal verification is GitHub Actions only. Local commands are development checks, not final acceptance.
- The one-click formal entry is `.github/workflows/goal.yml`.
- First-phase formal target platforms are Windows x86_64 and Linux x86_64.
- ARM64 remains experimental and non-blocking for now.
- Do not fork or copy GPL-3.0 code from `my-mibbrowser`.
- Do not add vendor MIB files or non-desensitized real walk data.

Additional `jacyl4/agent_skills` rules to apply:

- `karpathy`: prefer simple, surgical, verifiable changes; avoid speculative abstractions and unrelated cleanup.
- `browser-first-frontend-engineering`: keep Electron renderer browser costs and security boundaries explicit; avoid client-heavy complexity for a skeleton.
- `computer-first-performance-engineering`: avoid expensive hot-path patterns in shared scripts, but do not prematurely optimize cold skeleton code.

## Selected Approach

Use a small supervised Codex task. Claude remains responsible for scope, review, verification, and reporting. Codex writes the code for one bounded increment.

The increment is:

- Add `apps/desktop` as a workspace package.
- Add Electron main, preload, and React renderer entry points.
- Add Vite build configuration for renderer code.
- Expose a minimal safe preload API: `app.getVersion()`.
- Render a simple status page that calls the preload API and displays the app version.
- Add root scripts for desktop development/build checks only if needed by the new package.
- Keep current core parser/generator behavior intact.

## Non-Goals

This increment must not implement:

- SQLite database or migrations.
- Project management UI.
- SNMP profile forms.
- SNMP test connection.
- Online WALK / BULK WALK.
- MIB parser, OID matcher, switch classifier, or Chinese dictionary.
- Real installer packaging with electron-builder.
- GitHub Release creation.
- Broad refactors of existing `packages/core` or scripts.

## Architecture

`apps/desktop` should be split by Electron security boundary:

```text
apps/desktop/
  package.json
  index.html
  vite.config.ts
  tsconfig.json
  src/
    main/
      index.ts
    preload/
      index.ts
    renderer/
      main.tsx
      App.tsx
      global.css
    shared/
      preload-api.ts
```

Main process responsibilities for this increment:

- Create a browser window.
- Load Vite dev server URL in development.
- Load built renderer HTML in production-like builds.
- Enable `contextIsolation`.
- Disable `nodeIntegration`.
- Wire preload script.

Preload responsibilities:

- Use `contextBridge` to expose a minimal `window.zabtem.app.getVersion()` API.
- Avoid exposing raw Node or Electron primitives.

Renderer responsibilities:

- Use React to render a small skeleton page.
- Call `window.zabtem.app.getVersion()` and display the returned version.
- Keep state local and minimal.

Shared types:

- Define the preload API shape in one shared TypeScript file.
- Add global window typing for renderer usage.

## Data Flow

```text
Renderer React App
  -> window.zabtem.app.getVersion()
  -> preload contextBridge
  -> Electron app.getVersion()
  -> version string displayed in UI
```

No persistent data is introduced in this step.

## Error Handling

- If version lookup fails unexpectedly, the renderer should display a small fallback message rather than crash.
- Main-process startup failures can remain default Electron errors for this skeleton.
- No credentials or sensitive data are handled in this increment.

## Testing and Verification

Local development verification after Codex returns changes:

1. `corepack pnpm lint`
2. `corepack pnpm typecheck`
3. `corepack pnpm test`
4. `corepack pnpm generate:template`
5. `corepack pnpm validate:template`
6. `corepack pnpm build`
7. Any new desktop package build command added by the implementation, if not already covered by root `build`.

Formal acceptance remains a GitHub Actions `goal=full` run after changes are pushed.

## Codex Supervision Contract

Claude will give Codex this bounded task and constraints:

- Make only the desktop skeleton changes required by this spec.
- Follow existing TypeScript style and pnpm workspace patterns.
- Do not touch unrelated files except package/workspace/build config needed for the skeleton.
- Do not add GPL-derived code or copy from `my-mibbrowser`.
- Do not implement future product features.
- Add tests only where they provide meaningful verification for the skeleton.
- Return a concise summary of changed files and verification run.

Claude will then:

- Inspect the actual diff.
- Run verification locally using Corepack pnpm.
- Report failures honestly.
- Treat GitHub Actions as the only formal acceptance signal.

## Spec Self-Review

- No placeholders remain.
- Scope is limited to one implementation increment.
- The design does not contradict the repository plan: it starts with the desktop skeleton and keeps CI as formal acceptance.
- Ambiguous packaging expectations are resolved by explicitly excluding real installer packaging from this increment.
